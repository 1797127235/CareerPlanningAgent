# Plan: 模块串联 — 成长闭环

> 目标：把 JD 诊断、面试练习、学习路径、求职追踪、画像 五个孤岛模块串联成一条完整的成长链路，用户在任何一个模块的行为都能推动其他模块向前。

## 现状问题

当前各模块数据流是单向、断裂的：

```
JD 诊断 ──→ (gap_skills 产出，但无后续)
面试练习 ──→ (得分产出，但不影响画像)
学习路径 ──→ confirm-skill ──→ 画像技能更新 (唯一闭环)
求职追踪 ──→ (手动管理状态，不联动其他模块)
成长看板 ──→ (只读聚合，无反向推动)
```

用户体验：做完 JD 诊断后不知道下一步做什么；面试练完不知道自己哪里进步了；求职追踪和学习路径毫无关联。

## 目标闭环

```
                    ┌──────────────┐
                    │   JD 诊断     │
                    │  gap_skills  │
                    └──────┬───────┘
                           │ ① 差距 → 学习推荐
                           ▼
┌──────────┐      ┌──────────────┐      ┌──────────────┐
│ 求职追踪  │◄────│   学习路径     │────►│    画像       │
│ 投递/面试 │  ⑤  │  topic学习    │  ③  │  技能升级     │
└─────┬────┘      └──────┬───────┘      └──────┬───────┘
      │                  │                      │
      │ ④ 面试反馈       │ ② 学够了 → 模拟面试    │ ⑥ 画像变化 → 重新诊断
      │  → 薄弱点学习    │                      │
      ▼                  ▼                      │
┌──────────────┐  ┌──────────────┐              │
│  面试复盘     │  │  模拟面试     │──────────────┘
│  (debrief)   │  │  得分 → 技能  │  ③ 表现好 → 确认技能
└──────────────┘  └──────────────┘
```

## 串联点清单

### ① JD 诊断 → 学习路径推荐

**触发**：用户完成 JD 诊断后，结果页底部展示学习推荐。

**后端改动**：
- `jd_service.py` 的 `diagnose()` 已经产出 `gap_skills`
- 新增接口 `GET /graph/learning-suggestions?skills=skill1,skill2`
  - 输入：缺失技能列表
  - 输出：匹配的学习路径 topics（按相关度排序，最多 5 个）
  - 逻辑：遍历 `learning_paths.json`，找包含这些 skill 关键词的 topic，返回 `{ topic_id, title, title_zh, matched_skills[], subtopic_count }`

**前端改动**：
- JD 诊断结果页（`JdResultPage` 或诊断详情组件）底部新增"补齐差距"区块
- 每个推荐 topic 一张小卡片，点击跳转到学习路径页 `navigate('/learning-path?role=xxx&topic=yyy')`
- 如果没有匹配的 topic，不显示该区块

**设计稿**：
```
┌─ 诊断结果 ──────────────────────────┐
│ 匹配度 72%   技能差距 3 项            │
│ ▸ 缺少: Redis, 消息队列, K8s         │
├─────────────────────────────────────┤
│ 💡 补齐差距                          │
│ ┌─────────┐ ┌─────────┐ ┌────────┐ │
│ │ Redis    │ │消息队列   │ │ K8s    │ │
│ │ 3个子话题 │ │ 5个子话题 │ │4个子话题│ │
│ │ [去学习→]│ │ [去学习→]│ │[去学习→]│ │
│ └─────────┘ └─────────┘ └────────┘ │
└─────────────────────────────────────┘
```

### ② 学习路径 → 模拟面试引导

**触发**：用户在某个 topic 下完成全部 subtopic 后，弹出引导。

**后端改动**：
- `POST /graph/learning-path/progress` 返回值新增 `topic_completed: boolean`
- 当 `topic_completed=true` 时前端展示引导

**前端改动**：
- 学习路径页检测到 `topic_completed` 后，显示 toast/modal：
  ```
  🎉 「Redis」学完了！来一场模拟面试检验成果？
  [开始模拟面试]  [稍后再说]
  ```
- 点击"开始模拟面试"→ `navigate('/mock-interview?skill_tags=redis')`，自动筛选该 topic 相关的面试题

**无后端依赖**：面试路由已支持 `skill_tags` 参数。

---

### ③ 模拟面试 → 技能确认

**触发**：模拟面试结束，得分 ≥ 70 的维度对应的技能自动建议确认。

**后端改动**：
- `POST /mock/finish` 返回值新增 `confirmable_skills: string[]`
  - 逻辑：从 session 的 `skill_tags` + 问题涉及的技能中，筛选得分 ≥ 70 的
  - 只返回用户 profile 中尚未标记为 `familiar` 以上的技能

**前端改动**：
- 面试结果页底部展示"技能确认"区块：
  ```
  ✅ 本次面试表现不错！以下技能可以确认掌握：
  [Redis ✓]  [消息队列 ✓]
  [确认掌握]
  ```
- 点击确认 → 调用现有 `POST /graph/learning-path/confirm-skill`
- 确认后显示 toast（复用现有 toast 逻辑）

---

### ④ 面试复盘 → 薄弱点学习

**触发**：用户提交面试复盘（debrief）后，分析结果中的薄弱技能关联到学习推荐。

**后端改动**：
- `applications.py` 的 debrief 分析已有 `weak_areas` 字段
- 复用 ① 的 `GET /graph/learning-suggestions?skills=xxx` 接口

**前端改动**：
- 复盘结果页增加"针对性提升"区块
- 展示与 ① 相同的 topic 推荐卡片
- 文案改为"面试中这些方面可以加强"

---

### ⑤ 求职追踪 → 学习路径联动

**触发**：用户添加求职记录时，如果关联了 JD 诊断，自动展示该 JD 的差距技能进度。

**后端改动**：
- `GET /applications/{id}` 返回值新增 `gap_progress` 字段
  - 从关联的 JD 诊断取 `gap_skills`
  - 检查每个 gap skill 在 profile 中的当前状态（无/beginner/familiar/...）
  - 返回 `[{ skill: "Redis", status: "learned" }, { skill: "K8s", status: "gap" }]`

**前端改动**：
- 求职详情页展示"技能差距进度"条：
  ```
  技能准备 2/3 ████████░░░
  ✅ Redis  ✅ 消息队列  ⬜ K8s
  ```

---

### ⑥ 画像变化 → 引导重新诊断

**触发**：用户确认新技能后，如果之前做过 JD 诊断，引导重新诊断看匹配度变化。

**后端改动**：
- 在 guidance_service 的阶段判断中增加条件：
  - 如果 `last_skill_confirmed_at > last_jd_diagnosis_at`，引导消息改为"你的技能有更新，要不要重新诊断之前的 JD 看看匹配度变化？"
- 需要在 profile 或 stats 中追踪 `last_skill_confirmed_at`

**前端改动**：
- 首页 guidance banner 已支持动态消息，无需改动

## 优先级排序

| 优先级 | 串联点 | 复杂度 | 用户价值 | 依赖 |
|--------|--------|--------|----------|------|
| P0 | ① JD → 学习推荐 | 中 | 极高 | 新接口 |
| P0 | ② 学完 → 模拟面试 | 低 | 高 | 前端改动 |
| P1 | ③ 面试 → 技能确认 | 中 | 高 | finish 接口改动 |
| P1 | ⑥ 画像变化 → 重新诊断 | 低 | 中 | guidance 改动 |
| P2 | ④ 复盘 → 薄弱学习 | 低 | 中 | 复用①接口 |
| P2 | ⑤ 求职 → 差距进度 | 中 | 中 | 新字段 |

**建议实施顺序**：① → ② → ③ → ⑥ → ④ → ⑤

P0 两项（① + ②）可以并行开发，预计改动文件：
- 后端 2 文件：`graph.py`（新接口）、`mock.py`（返回值扩展）
- 前端 3 文件：JD 结果页、学习路径页、面试结果页

## 不在此次范围

- 成长看板页重构（独立 plan）

## 技术约束

- 不引入新依赖，所有串联用现有 API 模式（REST + rawFetch）
- 学习推荐接口做成通用的 `learning-suggestions`，JD 诊断和面试复盘共用
- 前端跳转用 query params 传递上下文（如 `?skill_tags=redis`），不用全局状态
- guidance 系统保持 stage-based，新增条件判断而非重写
