# Backend2 重构计划

他妈的啥也不懂就只会一直ai只能写出抽象不可维护的代码，到了后面ai自己也看不懂了！！！！

## 1. 背景

当前后端已经能支撑主要产品功能，但代码结构仍然带有原型期特征：功能跑通优先，模块边界和数据契约没有完全收住。继续在现有 `backend/` 上做大规模拆分，容易一边修一边破坏现有前端可用性。

因此，本次重构采用旁路重建策略：保留现有 `backend/` 作为稳定版本，新建 `backend2/` 逐步实现清晰边界、稳定契约和更适合开源维护的后端架构。

## 2. 核心目标

1. 收边界：每个业务能力有唯一主入口，避免 router、service、Graph 节点、tool 各自实现一遍。
2. 定契约：核心数据结构使用 Pydantic schema 明确版本和字段含义，减少散落的 `result_json` 隐式约定。
3. 减少重复入口：页面 API、LangGraph 编排、批处理能力都调用同一组业务 service。
4. 保持可运行：旧后端继续服务现有前端，新后端以 `/api/v2` 方式逐步接入。
5. 面向开源：让新人能跑起来、看懂架构、知道如何扩展模块。

## 3. 非目标

第一阶段不做以下事情：

- 不一次性替换现有 `backend/`。
- 不立即迁移全部数据库结构。
- 不强行把所有前端页面改成多 Agent。
- 不重写所有业务模块。
- 不为了目录好看做大范围无收益抽象。

## 4. 总体策略

采用“旁路重建、逐页切换、旧线冻结”的策略。

`backend/`：

- 保留现有功能。
- 只做必要 bug 修复。
- 不再承接大规模新架构能力。

`backend2/`：

- 从清晰目录和契约开始。
- 先做少量高价值模块。
- 通过 `/api/v2` 暴露接口。
- 前端逐页切换，切完一个模块再废弃旧接口。

## 5. 目录设计

建议第一版目录如下：

```text
backend2/
  app.py                    # FastAPI v2 app
  core/
    config.py               # 配置、环境变量
    errors.py               # 统一错误类型
    security.py             # 鉴权与用户上下文
  db/
    session.py              # DB session
    models.py               # v2 ORM 或旧模型适配
    repositories/           # 数据访问封装
  schemas/
    common.py               # 通用响应、分页、错误结构
    profile.py              # 用户画像契约
    opportunity.py          # 职位机会评估契约
    growth_plan.py          # 缺口行动计划契约
    evidence.py             # 成长证据契约
    interview_prep.py       # 面试准备契约
    report.py               # 阶段报告契约
  services/
    profile/
      context.py            # ProfileContext 构建
    graph/
      context.py            # RoleGraphContext 构建
    opportunity/
      parser.py             # JD 解析
      matcher.py            # 画像匹配
      evaluator.py          # 机会评级
      planner.py            # 下一步行动
      service.py            # 统一职位机会评估服务
    growth_plan/
      service.py            # 缺口行动计划
    evidence/
      service.py            # 项目、投递、面试、成果等证据沉淀
    interview_prep/
      service.py            # 面试重点、追问、STAR 故事
    report/
      service.py            # 阶段报告汇总
  routers/
    health.py
    profile.py
    opportunity.py
    growth_plan.py
    evidence.py
    interview_prep.py
    report.py
  llm/
    client.py               # 模型调用
    json.py                 # JSON 解析与修复
    prompts/                # prompt 模板
  graphs/
    assistant/
      state.py              # Assistant 图状态，只存 raw state
      nodes.py              # 解释、问答、自然语言入口
      tools.py              # LangGraph 工具适配，只调用 services
      graph.py              # 构建并导出 compiled graph
  langgraph.json            # LangGraph 应用配置，声明可部署图
  tests/
```

关键原则：`routers/` 不写业务逻辑，`services/` 承担核心业务，`graphs/` 只做后期可选的自然语言解释和跨模块编排。第一阶段不依赖 Assistant/Coach 也必须能跑通核心闭环。

## 6. 第一阶段范围

第一阶段做 Opportunity Core 最小闭环，而不是重做 Coach。目标是：不依赖聊天，也能完成一次从画像、JD 评估、行动计划到面试准备的核心求职链路。

第一阶段核心链路：

```text
ProfileContext
  -> Opportunity Evaluation
  -> Growth Plan
  -> Evidence
  -> Interview Prep
```

其中 `Opportunity Evaluation` 是第一阶段发动机，`Growth Plan`、`Evidence`、`Interview Prep` 先做轻量版本，用来证明闭环成立。

### 6.1 保留旧后端功能

旧接口继续可用：

```text
POST /api/jd/diagnose
GET  /api/jd/history
GET  /api/jd/{diagnosis_id}
POST /api/chat
```

### 6.2 新增 v2 接口

新增接口：

```text
POST /api/v2/opportunities/evaluate
GET  /api/v2/opportunities/evaluations
GET  /api/v2/opportunities/evaluations/{id}
POST /api/v2/growth-plans/from-evaluation/{evaluation_id}
POST /api/v2/interview-prep/from-evaluation/{evaluation_id}
```

第一版只要求页面可手动调用或通过测试验证，不急着切换前端。

## 7. Opportunity Evaluation 契约

第一版输出结构：

```json
{
  "schema_version": "opportunity_evaluation.v1",
  "evaluation_id": 1,
  "source": "page",
  "company": "",
  "jd_title": "",
  "role_summary": {
    "role_name": "",
    "seniority": "",
    "employment_type": "",
    "location_mode": "",
    "core_responsibilities": []
  },
  "match": {
    "score": 0,
    "level": "low | medium | high",
    "dimensions": {
      "basic": { "score": 0, "detail": "" },
      "skills": { "score": 0, "detail": "" },
      "qualities": { "score": 0, "detail": "" },
      "potential": { "score": 0, "detail": "" }
    },
    "matched_skills": [],
    "gap_skills": []
  },
  "opportunity": {
    "rating": "A | B | C | D",
    "type": "stretch | stable | practice | not_recommended",
    "recommendation": "",
    "reasoning": []
  },
  "evidence": [
    {
      "jd_requirement": "",
      "profile_evidence": "",
      "strength": "strong | medium | weak | missing"
    }
  ],
  "resume_strategy": [],
  "interview_focus": [],
  "posting_risks": [],
  "next_actions": [],
  "graph_context": null,
  "followups": {
    "growth_plan_id": null,
    "interview_prep_id": null,
    "report_id": null
  }
}
```

说明：

- `match.score` 保留现有 0-100 评分体系。
- `opportunity.rating` 用于回答“值不值得投”。
- `evidence` 用于把 JD 要求映射到用户真实项目、技能或教育经历。
- `interview_focus` 后续可直接喂给模拟面试模块。
- `next_actions` 后续可直接沉淀到 Growth Ledger。
- `schema_version` 必须保留，为后续兼容升级留出口。

## 8. 核心业务服务边界

### 8.1 Opportunity Evaluation

```text
OpportunityParser
  输入: jd_text
  输出: 公司、岗位、职责、要求、技能、经验、学历、软技能

OpportunityMatcher
  输入: parsed_jd, profile_context
  输出: 匹配分、维度分、匹配技能、缺口技能、证据链

OpportunityEvaluator
  输入: parsed_jd, match_result, graph_context
  输出: 机会评级、机会类型、投递建议

OpportunityPlanner
  输入: match_result, opportunity_result
  输出: 简历策略、面试重点、下一步行动

OpportunityEvaluationService
  统一编排以上能力，并负责保存与返回稳定契约
```

页面、Graph 节点、批处理只允许调用 `OpportunityEvaluationService`，不直接调用内部组件。

### 8.2 Growth Plan

把 `next_actions` 转成用户能执行的任务：

- 学习任务：补基础知识。
- 项目任务：形成可展示证据。
- 简历任务：把已有经历改成证据表达。
- 投递任务：跟进机会状态。

第一阶段只要求能从一次 evaluation 生成 3-5 个任务，不要求完整项目管理。

### 8.3 Evidence

Evidence 是核心闭环的沉淀层，不是普通日志。它记录：

- 这条证据来自哪个行动计划或哪次面试。
- 对应哪个岗位方向或 JD 缺口。
- 是否能转成简历 proof point。
- 是否能转成面试 STAR 故事。

第一阶段可以复用旧成长记录表，但 v2 输出必须有稳定 schema。

### 8.4 Interview Prep

基于 `OpportunityEvaluation` 和 `Evidence` 生成：

- 面试重点。
- 技术追问。
- 项目讲述角度。
- 行为面试 STAR 故事候选。

第一阶段先生成准备清单，不急着重做完整模拟面试。

## 9. LangGraph / Agent 边界

Backend2 中 LangGraph 不作为核心业务依赖。第一阶段核心业务不依赖 Assistant/Coach 也必须可运行。

后期如果保留 Assistant Graph，它只承担解释层和自然语言入口。Graph 节点只做三件事：

1. 识别用户意图。
2. 选择合适 service。
3. 把 service 的结构化结果转成对话表达。

Graph 节点和工具不应该：

- 直接查写数据库。
- 自己保存 `JDDiagnosis` 或 `CoachResult`。
- 自己拼业务 JSON。
- 自己实现 JD 匹配算法。

正确关系：

```text
前端页面 -> OpportunityEvaluationService / GrowthPlanService / EvidenceService / InterviewPrepService
Assistant Graph -> 上述 services
Report Graph 或 ReportService -> 汇总上述 services
```

状态设计遵循 LangGraph 的 raw state 原则：state 只保存消息、用户 ID、当前意图、结构化上下文 ID、服务返回的结构化结果，不保存拼好的 prompt、长篇上下文摘要或展示文案。prompt 在具体 node 中按需格式化。

## 10. 数据保存策略

第一阶段可以继续复用旧表，降低迁移成本：

- 继续使用 `profiles.profile_json` 读取用户画像。
- 可继续使用 `jd_diagnoses.result_json` 保存 v2 结果。
- 但写入时必须带 `schema_version = "opportunity_evaluation.v1"`。

后续再考虑新增表：

```text
opportunity_evaluations
  id
  user_id
  profile_id
  jd_text
  company
  role_title
  match_score
  opportunity_rating
  result_json
  schema_version
  created_at
```

原则：先稳定契约，再迁移存储。

## 11. 迁移路线

### Phase 0: 冻结旧后端边界

- 标记 `backend/` 为 legacy。
- 新功能优先进入 `backend2/`。
- 旧后端只修 bug。

交付物：

- `backend2/` 基础目录。
- `/api/v2/health`。
- README 或启动说明。

### Phase 1: Opportunity Core v1

- 实现 `OpportunityEvaluationService`。
- 实现 `POST /api/v2/opportunities/evaluate`。
- 输出稳定 `opportunity_evaluation.v1` 契约。
- 可复用旧 `JDService.diagnose` 作为底层能力，但外层契约必须用 v2。
- 实现轻量 `GrowthPlanService`，可从 evaluation 生成 3-5 个行动任务。
- 实现轻量 `InterviewPrepService`，可从 evaluation 生成面试准备重点。

交付物：

- Opportunity v2 接口。
- Growth Plan 最小接口。
- Interview Prep 最小接口。
- 单元测试覆盖空 JD、无画像、正常 JD、LLM 失败 fallback。
- 示例响应 JSON。

### Phase 2: 前端试点接入

- 在前端 2 新增隐藏或实验入口。
- 不替换旧 JD 页面。
- 对比旧 `/api/jd/diagnose` 和新 `/api/v2/opportunities/evaluate` 输出质量。

交付物：

- 一个实验页面或 dev-only 按钮。
- 旧接口与新接口并行可用。

### Phase 3: Evidence 接入

- `next_actions` 可转为成长任务和证据记录。
- 任务记录关联 `evaluation_id` 和 `gap_skill`。
- 用户完成任务后，可回写为简历证据或面试故事素材。

交付物：

- JD 缺口到行动任务再到证据沉淀的闭环。
- `EvidenceService` v2 雏形。

### Phase 4: Interview Prep 接入

- `interview_focus` 输入模拟面试模块。
- 面试结果回写到 JD evaluation 或 growth record。
- 沉淀 Story Bank。

交付物：

- 针对某份 JD 的面试准备链路。
- 面试反馈进入成长档案。

### Phase 5: Assistant Graph v2

- Assistant Graph 调用 v2 services。
- 不再通过 marker 和最近 DB 记录猜测卡片。
- 对话结果直接拿 service 返回的结构化 ID。

交付物：

- v2 assistant endpoint。
- LangGraph 编排与业务 service 解耦。

## 12. 测试策略

每个 v2 service 至少覆盖：

- 正常输入。
- 空输入。
- 用户画像缺失。
- LLM 返回非法 JSON。
- 缺少关键字段时的 normalize。
- schema_version 是否存在。

每个 v2 router 至少覆盖：

- 未登录返回 401。
- 无画像返回明确错误。
- 正常返回契约字段。
- 不泄露其他用户数据。

## 13. 开源准备要求

Backend2 从第一天就按开源项目标准组织：

- 有清晰启动命令。
- 有 demo 数据。
- 有 `.env.example`。
- 有 `schema_version` 和示例响应。
- 有隐私边界说明。
- 有最小测试集。
- 不把真实简历、真实 API key、私有数据放入仓库。

开源版最重要的不是功能最多，而是：

- 能跑起来。
- 能看懂模块边界。
- 能知道如何扩展新 Graph、Graph 节点或新 service。

## 14. 风险与约束

### 风险 1: 新旧后端长期并存

缓解：

- 每个模块切换完成后标记旧接口废弃。
- 文档记录当前前端页面使用哪个 API。

### 风险 2: v2 也重新变乱

缓解：

- 坚持 service 唯一入口。
- router 不写业务逻辑。
- Graph 节点和工具不直接查写 DB。
- 所有结果结构先写 schema。

### 风险 3: 数据迁移拖慢进度

缓解：

- 第一阶段复用旧表。
- 先稳定契约，再设计新表。

## 15. 第一周建议任务

1. 新建 `backend2/` 基础目录。
2. 跑通 `/api/v2/health`。
3. 定义 `schemas/opportunity.py`、`schemas/growth_plan.py`、`schemas/interview_prep.py`。
4. 实现 `services/opportunity/service.py` 的壳。
5. 复用旧 `JDService.diagnose`，包装为 `opportunity_evaluation.v1`。
6. 实现从 evaluation 生成 3-5 个行动任务的最小 `GrowthPlanService`。
7. 实现从 evaluation 生成面试准备重点的最小 `InterviewPrepService`。
8. 写最小测试：空 JD、无画像、正常 mock 结果、行动计划生成、面试准备生成。
9. 生成一份示例响应 JSON，放到 docs 或 tests fixtures。

## 16. 判断完成的标准

第一阶段完成标准：

- `backend2` 可以独立启动。
- `/api/v2/health` 可访问。
- `/api/v2/opportunities/evaluate` 可返回 `opportunity_evaluation.v1`。
- 不依赖 Assistant/Coach，也能完成“评估 -> 行动任务 -> 面试准备”的最小闭环。
- 旧 `backend/` 不受影响。
- 文档里能说明页面、Graph、service 的关系。
- 新增职位评估能力只需要改 `OpportunityEvaluationService`，不需要同时改 router、graph、tool 三处。
