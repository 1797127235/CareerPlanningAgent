# 系统设计面试知识点

### 高并发架构设计
水平扩展（Scale Out）通过增加节点分担负载；负载均衡算法（轮询、加权轮询、最少连接、一致性哈希）；网关设计统一入口，承担鉴权、限流、路由、协议转换；无状态服务设计便于弹性扩缩容。有状态服务处理：会话粘性、分布式会话、数据分片。

### 分布式一致性理论
CAP 定理：一致性、可用性、分区容错不可兼得，网络分区时通常选 CP 或 AP。BASE 理论：基本可用、软状态、最终一致。Raft 共识算法：Leader 选举、日志复制、安全性，实现相对简单（etcd、TiKV、Consul）。Paxos 理论完备但实现复杂。Gossip 协议用于成员发现与状态传播（Redis Cluster、Cassandra）。

### 分布式事务方案
2PC 两阶段提交：准备→提交，同步阻塞、单点故障（协调者）；3PC 增加 CanCommit 阶段，超时继续提交。TCC：Try/Confirm/Cancel，业务层实现补偿，空回滚与幂等处理。SAGA：正向/反向补偿，适合长事务（Seata Saga）。最终一致性：异步对账、消息队列保证可靠投递、最大努力通知。

### 缓存策略与问题
本地缓存（Caffeine、LRU Cache）低延迟但无共享；分布式缓存（Redis、Memcached）共享但增加网络开销。缓存穿透：查询不存在数据，用布隆过滤器或空值缓存；缓存击穿：热点 key 过期，用互斥锁或逻辑过期；缓存雪崩：大量 key 同时过期，用随机过期时间或多级缓存。缓存与数据库一致性：Cache-Aside、Read-Through、Write-Through、Write-Behind。

### 消息队列设计
Kafka：高吞吐、顺序磁盘写、零拷贝、Pull 模式、适合日志流；RocketMQ：事务消息、延迟消息、Push/Pull、适合电商金融。RabbitMQ：AMQP 协议、路由灵活、适合任务分发。消息可靠性：生产者确认（acks/all）、消费者手动 ACK、消息持久化、副本机制。顺序消费：同一 key 路由到同一分区。消息积压处理：扩容消费者、跳过非关键消息、降级。

### 数据库设计与优化
分库分表：垂直拆分按业务解耦，水平拆分按 key 分散数据；读写分离：主库写、从库读，主从延迟是核心挑战（延迟阈值、强制走主库）；索引优化：B+ 树结构、最左前缀、覆盖索引、避免索引失效；事务隔离级别：RU/RC/RR/SERIALIZABLE，InnoDB 默认 RR 通过 MVCC + Next-Key Lock 解决幻读。深度分页优化：游标分页、延迟关联。

### 可观测性体系
日志（Logging）：结构化日志、日志分级、集中采集（ELK/Loki/Fluentd）；指标（Metrics）：Counter/Gauge/Histogram，Prometheus + Grafana 监控与告警；链路追踪（Tracing）：分布式请求全链路追踪（Jaeger/SkyWalking/Zipkin），定位延迟瓶颈；告警设计：分级（P0-P3）、收敛（抑制/聚合/路由）、OnCall 轮值。错误预算：SLO 剩余预算决定发布节奏。

### 故障恢复与稳定性
熔断（Circuit Breaker）：Closed→Open→Half-Open，失败率触发；降级：关闭非核心功能、返回缓存数据/默认值、静态化页面；限流：令牌桶（匀速放行）、漏桶（平滑输出）、滑动窗口（精确计数），保护系统不被打垮；重试策略：指数退避、最大重试次数、防重试风暴（jitter 随机抖动）。舱壁隔离（Bulkhead）：线程池/信号量隔离，防止故障扩散。

### 分布式 ID 生成
雪花算法（Snowflake）：64 位结构（1位符号+41位时间戳+10位机器ID+12位序列号），时钟回拨问题与解决（等待/抛异常/缓冲）。号段模式（Leaf）：数据库批量取号段，内存中分配减少 DB 压力。UUID：无序、存储空间大、索引性能差。数据库自增：单点瓶颈、不适合分库分表。

### API 网关设计
网关核心职责：路由（动态路由/灰度路由）、鉴权（JWT/OAuth2/AKSK）、限流（分布式限流）、熔断、协议转换（HTTP/gRPC）、日志/监控。开源网关：Kong（OpenResty + Lua）、APISIX（ etcd 配置中心）、Spring Cloud Gateway（Reactor 异步）。性能优化：本地缓存热点配置、异步化、连接池。

### 服务注册与发现
服务注册：启动时注册到注册中心，心跳保活，下线时注销。服务发现：客户端发现（直连）vs 服务端发现（通过网关/代理）。注册中心：Consul（Raft + Gossip）、etcd（Raft）、ZooKeeper（ZAB）、Eureka（AP 设计，自我保护机制）。健康检查：主动探测（HTTP/TCP）vs 被动上报。

### 配置中心
配置集中管理，支持动态推送、灰度发布、版本回滚。Apollo：Namespace 隔离、灰度发布、权限控制。Nacos：配置 + 服务注册统一。Spring Cloud Config：Git 后端，Bus 推送。配置热更新：监听机制（长轮询/WebSocket/push），应用内刷新上下文。

### 搜索引擎
Elasticsearch：倒排索引、分片与副本、近实时搜索（refresh/translog）。与关系型数据库对比：不支持事务、最终一致性、适合全文检索和聚合分析。性能优化：合理分片数、冷热分离、索引生命周期管理（ILM）。

### 大数据处理
批处理：MapReduce（分治+归并）、Spark（内存计算、DAG 调度、RDD/DataFrame/Dataset）。流处理：Storm（低延迟）、Flink（有状态计算、Exactly-Once、窗口语义）、Spark Streaming（微批）。Lambda 架构（批流分离）vs Kappa 架构（统一流处理）。

### 容器与编排
Docker：Namespace（隔离）+ Cgroup（资源限制）+ UnionFS（镜像分层）。Kubernetes：Pod（最小调度单元）、Deployment（无状态）、StatefulSet（有状态）、Service（服务发现）、Ingress（七层路由）。调度：资源请求/限制、亲和性/反亲和性、污点/容忍。弹性伸缩：HPA（水平 Pod 自动伸缩）、VPA（垂直）、Cluster Autoscaler（节点级别）。

### 安全架构
零信任架构：永不信任，始终验证。最小权限原则。WAF（Web 应用防火墙）：规则引擎+机器学习识别攻击。DDoS 防护：流量清洗、黑洞路由、CDN 分散。数据安全：加密（传输 TLS/存储 AES）、脱敏、审计日志。安全开发生命周期（SDL）：需求→设计→编码→测试→发布各阶段安全检查。

### 容量规划
QPS/TPS 估算：日活×人均请求数÷峰值系数÷86400。峰值倍数：通常按 3-5 倍设计。资源瓶颈：CPU（计算密集型）、内存（缓存/大数据）、磁盘 IO（数据库/日志）、网络带宽（视频/文件）。压测方法论：基准压测→阶梯加压→极限探测，关注长尾延迟（P99）而非均值。

### 异地多活与容灾
RTO（恢复时间目标）与 RPO（恢复点目标）。同城双活：两个机房同时提供服务，数据库跨机房同步（延迟可控）。异地多活：单元化架构（用户路由到固定单元）、数据同步（Canal/Otter/Binlog）。灾备演练：定期切换演练、数据一致性校验、回滚流程验证。

### 面试追问模板
- 为什么这么分层？瓶颈先出在哪里？
- 失败后怎么恢复？谁触发补偿？如何防重？
- 峰值翻 3 倍时，先扩哪个组件？依据是什么？
- 你怎么证明方案有效？给出核心指标与阈值。
- 一致性和可用性冲突时，你的取舍原则是什么？
- 网络分区发生时，你的系统选择保 CP 还是 AP？为什么？
