# 测试开发面试重点（测试理论 + 自动化 + 性能 + CI/CD + 质量保障）

## 测试基础
- 软件测试定义与目的：发现缺陷、验证需求、提供质量反馈，不只是"找 Bug"。
- 测试流程：需求分析→测试计划→测试设计→测试执行→缺陷跟踪→测试报告→回归测试。
- 测试左移（Shift-Left）：需求/设计阶段就介入质量活动，早发现早修复，缺陷成本随阶段指数增长。
- 测试覆盖率：行覆盖/分支覆盖/路径覆盖/条件覆盖，高覆盖率 ≠ 高质量测试。
- 测试金字塔：单元测试（>70%）→服务/集成测试→UI/E2E 测试（<10%），投资回报递减。
- 测试右移（Shift-Right）：生产环境监控、混沌工程、用户反馈驱动的质量改进。

## 测试用例设计
- 等价类划分：有效/无效等价类，每类取代表性数据，减少冗余。
- 边界值分析：等价类边界最易出错，如范围 [1,100] 测试 0/1/100/101。
- 判定表法：多条件组合场景（如会员×节假日×提前天数→价格）。
- 场景法：模拟用户操作流程（主流程+异常分支），流程图覆盖。
- 错误推测法：基于经验推测易错点（空文件/超大文件/病毒文件/格式不符）。
- 状态迁移法：有状态变化对象（订单状态、电梯状态），验证状态转换。
- 正交实验法：多因素多水平组合，用最少的用例覆盖最大的组合空间。
- 好的测试用例标准：覆盖全面、可重复执行、独立无依赖、可维护。

## 测试类型
- 黑盒测试 vs 白盒测试：关注外部行为 vs 关注内部代码逻辑。
- 单元测试：最小可测试单元（函数/类），Mock 外部依赖，开发编写，CI 自动执行。
- 集成测试：多模块/服务交互验证，接口契约、数据传递、依赖协同。
- 系统测试：完整系统验证，功能/性能/安全/兼容性/易用性全面覆盖。
- 回归测试：修改后确认旧功能未被破坏，高度依赖自动化用例。
- 验收测试（UAT）：业务方确认是否满足需求，贴近真实场景。
- 端到端测试（E2E）：完整业务链路验证，从用户入口到最终输出。
- 探索性测试：同时设计和执行测试，依赖测试人员经验和直觉。
- 冒烟测试：快速验证核心流程是否可通，作为详细测试的准入条件。

## 自动化测试
- ROI 公式：(节省手工时间 × 执行频率) - (开发时间 + 维护成本)。
- 适用场景：回归测试、步骤确定、执行频率高；不适用：UI 频繁变动、探索性测试。
- Web 自动化：Selenium（多语言/多浏览器）、Cypress（JS-only）、Playwright（现代化/多语言/快）。
- 元素定位策略：ID > Name > CSS Selector > XPath，避免使用绝对路径。
- 等待机制：隐式等待 vs 显式等待（ExpectedConditions）vs 流畅等待（FluentWait）。
- 页面对象模型（POM）：封装页面元素和操作，提高可维护性。
- API 测试：REST/GraphQL/gRPC，状态码校验、响应体 Schema 校验、数据库断言。
- API 测试工具：Postman/Reqable/Insomnia，结合 Newman/CI 自动化回归。
- 移动端自动化：Appium（跨平台）、Espresso（Android 原生）、XCTest（iOS 原生）。
- 测试数据管理：数据池、数据工厂、数据库快照、Docker 化测试环境。

## 性能测试
- 压测类型：负载测试（预期负载验证）、压力测试（超负载找极限）、稳定性测试（长时间运行查泄漏）。
- 压测流程：确定目标（QPS/RT/TPS）→准备环境→选工具→编写脚本→执行→监控分析→调优→复测。
- 压测工具：JMeter（功能全面）、k6（JS 脚本）、Gatling（Scala）、Locust（Python）。
- 关键指标：TPS/QPS、响应时间（P50/P95/P99）、错误率、吞吐量。
- 资源监控：CPU/内存/磁盘 IO/网络带宽/连接数/GC 停顿。
- 瓶颈定位：数据库慢查询、连接池打满、线程池饱和、网络带宽、缓存命中率。
- 全链路压测：生产环境流量复制/影子库、风险隔离、数据清理。

## CI/CD 与质量门禁
- 流水线设计：代码提交→静态检查→单元测试→构建→集成测试→部署→E2E 测试→发布。
- 质量卡点：代码覆盖率阈值、SonarQube 质量门、自动化测试通过率、安全扫描。
- 持续集成：Jenkins/GitLab CI/GitHub Actions，自动触发构建与测试。
- 持续部署：蓝绿部署、金丝雀发布、滚动更新、特性开关（Feature Toggle）。
- 测试报告：Allure/ReportPortal，失败用例分类、趋势分析、 flaky test 识别。
- 失败告警：分级告警（P0-P3）、通知渠道（邮件/钉钉/企微/飞书）、OnCall 机制。

## 质量保障体系
- 质量度量：缺陷密度、缺陷逃逸率、测试覆盖率、发布频率、MTTR（平均恢复时间）。
- 缺陷管理：缺陷生命周期（新建→确认→分配→修复→验证→关闭），严重程度与优先级。
- 根因分析：5 Why、鱼骨图、缺陷分类（需求/设计/编码/配置/环境）。
- 风险预警：线上监控（错误率/延迟/流量突增）、自动熔断、降级演练。
- 混沌工程：Chaos Mesh/Litmus，故障注入（网络延迟/节点宕机/依赖不可用），验证系统韧性。
- 质量文化：质量是团队责任（而非测试部门独有），开发自测、Code Review、结对编程。

## 安全测试
- OWASP Top 10：XSS/SQL 注入/CSRF/身份认证失效/敏感信息泄露/访问控制失效等。
- XSS 测试：输入 `<script>alert(1)</script>`、`<img src=x onerror=alert(1)>`，验证转义/过滤。
- SQL 注入测试：`' OR '1'='1 --`、`' UNION SELECT --`、时间盲注 `AND SLEEP(5)`。
- CSRF 测试：检查 CSRF Token、SameSite Cookie、Referer 校验。
- 越权测试：水平越权（同角色他人数据）、垂直越权（普通用户访问管理接口）。
- 敏感信息泄露：日志脱敏、配置文件加密、接口返回字段过滤、HTTPS 强制。

## Bug 处理与协作
- 不可复现 Bug：多次尝试不同环境/数据→收集详细信息→与开发协作分析→增加日志→监控观察。
- 缺陷报告要素：标题、前置条件、复现步骤、预期结果、实际结果、严重程度、优先级、截图/日志。
- Bug 生命周期：新建→确认→分配→修复→验证→关闭（或重新打开）。
- 印象深刻的 Bug：场景描述 + 根因分析 + 修复过程 + 后续预防。
- 与开发沟通：客观描述现象而非指责、提供最小复现步骤、共同定位根因。

## 面试追问模板
- 登录/支付/电商直播间场景，你怎么设计测试用例？
- 你的自动化测试覆盖率是多少？ROI 怎么评估？
- 线上出了一个 Bug，但测试环境无法复现，你怎么排查？
- 你在 CI/CD 中集成了哪些质量门禁？怎么防止质量回退？
- 压测发现 QPS 上不去，你的排查思路是什么？
- 一个需求时间紧测试时间不够，你怎么保证质量？
